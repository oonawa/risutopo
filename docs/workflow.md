# 作業フロー

すべての作業は以下の4フェーズで進める。ユーザーの承認なしに次フェーズへ進まない。

## フェーズ 1: テスト計画

1. 要件を整理し、以下の内容を `TODO.md` に書き出す。

   ### テストシナリオ
   | # | シナリオ | 期待値 |
   |---|---------|--------|
   | 1 | ○○の場合 | ○○になる |

   ### テスト構成
   | 項目 | 内容 |
   |------|------|
   | テスト種別 | Medium / E2E |
   | 使用ツール | Vitest / Playwright |
   | モック対象 | Cookie / 外部API など |

2. ユーザーにテスト計画を提示し、承認を得る。
3. 承認されるまで計画を練り直す。承認なしにフェーズ 2 へ進まない。

## フェーズ 2: 実装計画

1. テスト計画の承認後、実装作業を `TODO.md` に追記する。
2. ユーザーに実装計画を提示し、承認を得る。
3. 承認されるまで計画を練り直す。承認なしにフェーズ 3 へ進まない。

## フェーズ 3: 実装

1. `TODO.md` を上から順に実行する。
2. **TDD で進める。**
   1. フェーズ 1 で承認されたテスト計画をコードとして書く。
   2. テストが失敗することを確認する（Red を飛ばさない）。
   3. テストが通る最小実装を書く。
3. **不具合修正も TDD で進める**（新規機能と同じ重要度）。
   - Red フェーズ：テストを先に書き、失敗を確認してからコード修正に入る。
   - 型チェックエラーを Red の代替として使用できるが、元となるテストケースを TODO.md に記載し、その意図を明確にすること。
4. リファクタリングは新規テスト不要。既存テストが通り続けることを保証する。
5. 複雑な判断が必要な局面ではモデルをエスカレーションしてよい。

## フェーズ 4: 検証

1. 編集したファイルに対して `npm run lint` を実行する。
2. プロジェクト全体の型チェックを実行する（`npx tsc --noEmit`）。
   - import の伝播で編集外ファイルがエラーになる場合があるため、型チェックは全体に対して行う。
3. テストを実行する（`npm run test && npx playwright test (今回追加 or 変更した /tests/e2e/* のテストファイル)`）。
4. テストが落ちた場合:
   - typo など明らかな軽微なミスは自動修正する。
   - それ以外は原因を調査してユーザーに報告し、修正プランの承認を待つ。

## TODO.md の扱い

- `.gitignore` 済みの個人ファイル。
- タスクごとに使い捨て。次のタスク開始時に上書きしてよい。
