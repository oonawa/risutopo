# 作業フロー

すべての作業は以下の3フェーズで進める。ユーザーの承認なしに計画フェーズを飛ばしてコード編集を始めない。

## フェーズ 1: 計画（Opus）

1. `/model opus` でモデルを切り替える。
2. 要件を整理し、作業計画を `TODO.md` に書き出す。
3. ユーザーに計画を提示し、承認を得る。
4. 承認されるまで計画を練り直す。承認なしにフェーズ 2 へ進まない。

## フェーズ 2: 実装（Sonnet）

1. `/model sonnet` でモデルを切り替える。
2. `TODO.md` を上から順に実行する。
3. **リファクタリング以外のコード編集は TDD で進める。**
   1. 仕様をテストコードとして書く。
   2. テストが失敗することを確認する（Red を飛ばさない）。
   3. テストが通る最小実装を書く。
4. リファクタリングは新規テスト不要。既存テストが通り続けることを保証する。
5. 複雑な判断が必要な局面では `/model opus` にエスカレーションしてよい。

## フェーズ 3: 検証

1. 編集したファイルに対して `npm run lint` を実行する。
2. プロジェクト全体の型チェックを実行する（`npx tsc --noEmit`）。
   - import の伝播で編集外ファイルがエラーになる場合があるため、型チェックは全体に対して行う。
3. 全テストを実行する（`npm run test`）。
   - 関連テストだけで満足するとリグレッションを見逃す。
4. テストが落ちた場合:
   - typo など明らかな軽微なミスは自動修正する。
   - それ以外は原因を調査してユーザーに報告し、修正プランの承認を待つ。

## TODO.md の扱い

- `.gitignore` 済みの個人ファイル。
- タスクごとに使い捨て。次のタスク開始時に上書きしてよい。
